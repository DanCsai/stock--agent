## Context

当前系统已经完成用户登录、资料、头像和按用户隔离的聊天基础能力，但聊天内容仍然主要依赖占位回复。为了让产品开始提供真实业务价值，需要引入基金搜索、基金详情和基金走势查询能力，并把这些能力组织成可被网页端和后续飞书机器人共同复用的服务层。

这个变更不仅会新增前端展示组件，还会影响现有问答接口、后端服务分层和外部数据源接入策略。当前阶段的重点不是直接做智能推荐，而是先建立一个稳定、可复用、可扩展的基金数据查询模块。

## Goals / Non-Goals

**Goals:**
- 支持基金代码精确查询和基金名称模糊查询。
- 在基金名称模糊查询时返回前 5 个候选基金。
- 支持查询基金基本信息，包括名称、代码、类型、基金公司、最新净值和日涨跌。
- 支持查询基金走势，并允许按天、月、3个月、半年和 1 年切换时间范围。
- 在前端使用“聊天简短文字 + 基金卡片 + 走势图 + 推荐/分析占位”的方式展示基金信息。
- 在后端抽象可复用的基金数据服务，为网页和后续飞书接入提供统一能力。

**Non-Goals:**
- 不在本次变更中实现真正的 Agent 推荐逻辑。
- 不在本次变更中实现基金优劣评估、多基金深度对比或板块策略分析。
- 不在本次变更中实现飞书机器人接入本身，只预留服务抽象和接口复用能力。
- 不依赖第三方网站直接嵌入走势图，而是基于真实数据自行渲染图表。

## Decisions

### 1. 使用真实基金数据源，并在后端抽象统一 Provider 接口

本次变更将接入真实基金数据源，而不是继续使用 mock 数据。为了避免将网页逻辑与单一数据源耦合，后端需要引入统一的基金数据提供方接口，封装：

- 基金搜索
- 基金基础信息
- 基金走势

这样后续可以在不影响上层调用方的情况下替换或增加数据源，也能让飞书入口直接复用现有服务。

考虑过的替代方案：
- 先使用 mock 数据快速上线
- 未采用，因为当前目标已经明确需要真实查询能力

### 2. 搜索结果先走“候选列表 + 用户选择”模式

基金名称模糊搜索会返回前 5 个候选基金，候选项中至少包含基金名称、代码和类型。对于唯一代码输入，可以直接查询详情；对于名称输入，优先展示候选列表，再进入详情查询。

这样既能提升可用性，也能减少名称冲突导致的误查。

考虑过的替代方案：
- 输入名称后直接取第一个结果
- 未采用，因为基金名称重复和近似命名较常见，容易误导用户

### 3. 前端采用“聊天文本 + 卡片 + 图表 + 占位区”组合展示

聊天区继续作为主交互入口，但基金查询结果不再只返回一大段文本，而是拆成结构化展示：

- 聊天中的简短说明
- 基金基础信息卡片
- 走势展示区域
- 推荐占位
- 分析占位

这样一方面提升信息可读性，另一方面为后续推荐和分析能力接入预留固定区域。

考虑过的替代方案：
- 只在聊天气泡里返回全部文本内容
- 未采用，因为基金信息和走势更适合结构化展示

### 4. 走势数据自行渲染，而不是嵌入第三方网站图表

前端图表应基于后端返回的真实走势数据自行渲染，而不是直接搬用第三方站点的走势图。这样做可以：

- 保持展示风格统一
- 控制性能与稳定性
- 为飞书等非网页入口复用同一数据结构

考虑过的替代方案：
- 直接嵌入第三方网站图表
- 未采用，因为可控性和长期稳定性较差

### 5. 将基金查询能力设计成可被多入口复用的服务

网页端只是当前第一个消费方。后端应把基金查询逻辑设计成独立 service，并将对话解析或入口适配与基金数据查询本身分离。后续飞书机器人只需要负责：

- 接收消息
- 识别基金查询意图
- 调用相同的基金服务
- 返回适配飞书格式的结果

考虑过的替代方案：
- 在网页 handler 中直接拼接数据源请求逻辑
- 未采用，因为这会阻碍后续多入口扩展

## Risks / Trade-offs

- [真实基金数据源可能存在调用频率、权限或稳定性限制] → 缓解方式：通过 Provider 抽象隔离数据源，必要时支持替换或降级。
- [基金名称模糊查询结果较多，容易增加用户选择成本] → 缓解方式：限制前 5 个候选，并在列表中展示名称、代码和类型。
- [前端增加卡片和走势图后，页面复杂度会上升] → 缓解方式：保持卡片字段克制，走势图先做清晰的单图层版本。
- [网页和飞书对展示形式要求不同] → 缓解方式：后端统一返回结构化基金数据，入口层自行格式化。

## Migration Plan

1. 引入基金数据 Provider 抽象，并接入第一版真实基金数据源。
2. 增加基金搜索、基金详情和基金走势的服务层。
3. 调整现有聊天接口，使其支持基金查询结果的结构化返回。
4. 在前端增加基金候选列表、基金信息卡片和走势图展示区域。
5. 增加推荐占位和分析占位区，但保持内容为占位信息。
6. 验证代码查询、名称查询、走势切换以及结构化展示链路。

由于本次变更是对现有聊天系统的增强，回滚策略相对直接。如果基金数据源接入不稳定，可以临时关闭基金查询能力并退回占位回复。

- 推荐占位和分析占位第一版是否只展示静态文案，还是返回结构化占位字段？
