## Context

当前基金助手 MVP 已经具备聊天页面、历史会话、后端接口和基础存储能力，但所有数据仍然是全局共享的。随着产品从单人演示逐步走向更真实的使用场景，用户身份、用户资料和数据隔离已经成为最优先需要补齐的基础能力。

这次变更不仅影响前端页面，还会影响数据模型、存储结构、认证流程以及聊天接口的访问范围，因此需要先通过设计文档明确边界和实现思路。目标是在保持现有 MVP 结构尽量简单的前提下，为“每个用户拥有独立资料和独立会话”建立清晰基础。

## Goals / Non-Goals

**Goals:**
- 增加注册和登录能力，支持账号、密码和用户名。
- 增加用户资料能力，支持默认头像和更换头像。
- 将现有聊天会话与消息绑定到具体用户，实现按用户隔离。
- 在前端提供注册页、登录页以及登录后的用户状态展示。
- 提升聊天主界面的视觉质量，优化聊天气泡尺寸、间距、信息层次和整体布局观感。
- 使用 MySQL 存储用户、资料、会话和消息数据。

**Non-Goals:**
- 本次变更不引入第三方 OAuth 登录能力。
- 不实现找回密码、短信验证、邮箱验证或复杂风控。
- 不实现云端对象存储头像上传流程。
- 不在本次变更中实现复杂的多环境数据库迁移体系或生产级高可用方案。

## Decisions

### 1. 使用 MySQL 存储用户、会话和消息结构

为了避免用户、会话和消息关系继续堆积在本地文件中，这次将把核心业务数据切换到 MySQL 中管理。数据库结构至少覆盖：

- 用户列表
- 用户资料
- 用户与会话的绑定关系
- 会话与消息

这样可以让多用户数据隔离更清晰，也为后续查询、扩展和管理提供更稳定的基础。

考虑过的替代方案：
- 继续沿用本地文件存储
- 未采用，因为用户、资料、会话和消息一旦同时增长，文件结构会迅速变复杂，不利于后续扩展

### 2. 使用服务端会话或本地 token 维持登录态

为了让前端在登录后能持续访问自己的会话，系统需要一个最小可用的登录态机制。MVP 阶段优先采用简单、可控的登录态方案，例如服务端生成会话标识并通过 cookie 或轻量 token 返回给前端。

这样可以让现有会话接口在处理请求时识别当前用户，并为后续更完整的认证体系保留演进空间。

考虑过的替代方案：
- 先不做登录态，每次请求都传用户名
- 未采用，因为这无法真正实现身份校验和用户隔离

### 3. 密码以摘要形式存储，而不是明文存储

虽然当前只是 MVP，但密码仍然不应以明文方式写入数据库。系统应保存密码摘要，注册和登录时通过摘要比对完成校验。

考虑过的替代方案：
- 直接明文保存密码，加快实现
- 未采用，因为即便是本地原型，也会形成明显的安全反模式

### 4. 头像先采用默认头像 + 文本路径或资源引用的简化方案

本次变更会支持默认头像，并允许用户更换头像，但不引入复杂上传链路。更换头像可以先基于预置头像、图片路径或简单资源引用完成。

这样既能满足资料展示与界面美观需求，也能避免在本次变更中额外解决文件上传、校验和存储安全问题。

考虑过的替代方案：
- 直接实现图片上传和静态资源管理
- 未采用，因为这会显著扩大本次变更范围

### 5. 聊天页面在登录后展示，并包含当前用户信息区与更精致的消息布局

前端需要从“只有聊天页”升级为“登录/注册 + 登录后聊天页”的产品流。聊天页应显示当前用户信息，例如用户名和头像，并在界面结构上预留资料入口。同时需要优化消息气泡的最大宽度、内边距、对齐方式和留白层次，避免当前消息框过大、阅读压力过高的问题。

考虑过的替代方案：
- 只做后端用户体系，不改前端交互
- 未采用，因为用户体系如果不进入前端体验，无法完成完整验证

### 6. UI 美化先聚焦布局、比例和信息层次，不追求一次性做完所有视觉系统

本次 UI 优化重点应放在：

- 登录/注册页的可用性和观感
- 聊天气泡宽度和文本可读性
- 头像、用户名和操作入口的层次
- 左侧会话列表与右侧对话区的比例

这样可以先解决当前“对话框过大、界面过空、信息层次不够清楚”的问题，同时避免把这次变更扩大成完整设计系统重构。

考虑过的替代方案：
- 一次性全面重做所有页面视觉规范
- 未采用，因为这会让本次需求从“补齐用户体系”膨胀成一个过大的设计工程

## Risks / Trade-offs

- [MySQL 接入会让当前 MVP 比纯本地存储更复杂] → 缓解方式：先控制表结构数量，优先满足用户、会话、消息和资料四类核心数据。
- [登录态方案如果过于简化，后续可能需要重构] → 缓解方式：把认证逻辑集中在单独模块，避免直接散落到各个 handler。
- [头像方案过于简化，未来可能不满足真实上传需求] → 缓解方式：本次先把“头像字段和更新入口”建好，后续再替换底层实现。
- [聊天接口从匿名访问切到用户访问后，旧数据处理会有兼容问题] → 缓解方式：明确旧的全局聊天数据仅服务 MVP 原型，并在实现时定义迁移或清理策略。
- [UI 调整如果没有明确约束，容易继续出现气泡过大或布局空洞的问题] → 缓解方式：在前端任务中明确限制消息区域宽度、组件间距和用户信息区布局。

## Migration Plan

1. 设计并建立 MySQL 表结构，增加用户、资料和用户范围内的会话关系。
2. 增加注册、登录和当前用户信息接口。
3. 调整聊天相关接口，使其基于当前登录用户读写数据。
4. 新增前端登录页、注册页和登录后用户信息区。
5. 调整聊天页面样式和布局，使其展示用户名和头像，并优化消息气泡尺寸与阅读体验。
6. 验证注册、登录、会话隔离和头像更新的完整流程。

由于当前仍处于本地 MVP 阶段，回滚策略相对直接。如果这次变更需要暂停，可以先保留已有聊天原型，并移除登录入口或用户隔离逻辑。

## Open Questions

- MVP 第一版登录态应优先采用 cookie 还是简单 token？
- 默认头像是使用单一默认图，还是提供一组预设头像可选？
- 用户名是否允许后续修改，还是先固定为注册后不可改？
- 旧的全局聊天数据是否需要迁移到某个默认用户，还是直接清空重建？
