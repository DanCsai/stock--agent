## Context

项目当前几乎还是一个空仓状态，需要先建立一个可落地的基金推荐助手第一阶段里程碑。目标技术栈已经比较明确，包括聊天式前端和基于 Hertz 的 Go 后端。团队当前更希望优先打通完整链路，同时保留足够的扩展性，方便后续继续叠加模型能力、基金分析和股票分析能力。

第一版用户体验会有意控制范围，先只提供会话面板、历史会话以及一个能够返回占位回答的后端问答接口。这个设计强调 UI、API、回复服务和存储层之间的清晰边界，以便 MVP 能够快速上线，同时不阻碍后续扩展。

## Goals / Non-Goals

**Goals:**
- 建立前端聊天流程，支持创建会话、查看历史会话和展示消息内容。
- 定义后端接口，覆盖会话管理与助手问答提交。
- 引入一个最小可运行的回复服务，接收用户输入并返回占位文本回答。
- 持久化保存会话和消息，使对话能够被恢复和继续。
- 保留一个可扩展架构，后续可以在不重做整体结构的前提下增加模型能力、基金数据工具、推荐逻辑和股票支持。

**Non-Goals:**
- 本次变更不交付生产级别的基金推荐逻辑。
- MVP 阶段不集成实时行情数据、图表组件或板块看板。
- 不支持强交易指令或高确定性的投资决策建议。
- 不在本次变更中接入大模型、Eino 或 Agent 工具调用能力。
- 不在本次变更中确定长期模型选型、评估体系或金融推荐合规策略。

## Decisions

### 1. Separate the system into frontend, API, reply service, and persistence layers

MVP 将采用四层结构：

- 前端：聊天页面与历史会话界面
- API：基于 Hertz 的会话与消息接口处理层
- 回复服务：用于生成占位回答或规则化回答
- 存储层：负责会话与消息持久化

这样可以把用户交互逻辑和回复生成逻辑清晰分离，避免前端直接依赖具体实现，也为后续替换成 LLM/Agent 版本预留空间。

考虑过的替代方案：
- 使用一个单体后端 handler，直接调用模型，不做服务边界拆分
- 未采用，因为虽然前期更快，但会让后续的模型接入、测试和存储演进更困难

### 2. Start with a stub reply service instead of integrating LLM or agent runtime

第一版会把助手实现成一个简单的回复服务，先以固定模板或轻量规则生成回答为主。后端会把用户输入和有限的上下文传给回复服务，并返回生成结果。

这样既能保留后续演进成 LLM/Agent 的方向，也能避免在模型接入、提示词设计和工具流程尚未明确前过早设计。

考虑过的替代方案：
- 在 MVP 中直接接入 Eino 和大模型
- 未采用，因为当前阶段更需要先验证前后端流程与历史会话能力

### 3. Store conversations explicitly instead of relying on stateless chat only

系统会显式存储会话元数据和有序消息历史。这既支撑历史会话 UI，也方便后端在后续接入模型能力时带入近期上下文。

考虑过的替代方案：
- 使用无状态问答接口，只在前端本地保留历史记录
- 未采用，因为这会破坏多会话连续性，也会让后续分析能力和 Agent 记忆能力更难实现

### 4. Design the domain around “assistant” semantics, not only “fund” semantics

虽然 MVP 聚焦基金，但 API 和回复服务的边界不应该过度绑定到某一种资产类型。整体契约应该围绕会话、消息和回答来设计，基金相关能力更多放在后续规则、模型或领域服务中。

考虑过的替代方案：
- 直接把基金推荐规则硬编码进 API 契约
- 未采用，因为团队已经明确后续希望扩展到股票场景

### 5. Defer advanced financial advice and intelligence behind clear scope boundaries

MVP 会支持基本的占位回答和消息流转验证，但不会提供真实的基金推荐结论，也不会宣称确定性的投资决策。这样可以在验证产品流程期间控制风险，并把智能能力留到后续阶段。

考虑过的替代方案：
- 从第一天就把 MVP 定义成带智能推荐能力的助手
- 未采用，因为项目当前还缺少稳定的数据源、模型接入方案、评估体系和合规保护措施

## Risks / Trade-offs

- [用户可能过早期待智能推荐能力] → 缓解方式：在 MVP 阶段明确第一版仅提供聊天骨架、历史会话和占位回复。
- [后续引入模型能力时，当前回复服务可能需要替换] → 缓解方式：提前保持回复服务与 API 的边界清晰，避免前端依赖具体实现。
- [当后续需要更丰富元数据时，会话存储设计可能需要调整] → 缓解方式：从一开始就保留可扩展的 session/message 结构，为角色、工具轨迹和扩展字段预留空间。
- [项目仍在收敛阶段，前后端接口可能发生漂移] → 缓解方式：在真正开始实现前，先通过 spec 把核心会话与消息接口定义清楚。

## Migration Plan

1. 搭建前端聊天页面和基础页面结构。
2. 搭建基于 Hertz 的后端服务骨架，并提供健康检查与助手接口。
3. 增加适合本地开发的简易存储，用于保存会话与消息。
4. 将助手问答接口接入最小化的占位回复服务。
5. 打通前后端消息链路，并验证历史会话加载流程。
6. 在后续变更中继续补充模型能力、更丰富的基金数据和工具型 Agent 能力。

由于这是一组全新的能力，回滚策略相对简单。如果中途需要暂停实现，可以直接关闭聊天入口或助手问答接口，而不会影响已有线上功能。
